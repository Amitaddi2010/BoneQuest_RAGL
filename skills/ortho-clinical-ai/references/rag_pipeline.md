# BoneQuest RAG Pipeline Reference

## 3-Signal Hybrid Retrieval Architecture

BoneQuest uses a three-signal fusion approach for document retrieval:

### Signal 1: BM25 Keyword Search
- **Purpose**: Exact keyword matching for precise medical terminology
- **Strengths**: Excellent for specific terms like "Gustilo-Anderson Type IIIB", classification names, drug names
- **Backend**: Whoosh-based inverted index
- **When dominant**: User queries with specific medical jargon

### Signal 2: Semantic Embedding Search
- **Purpose**: Meaning-based similarity matching
- **Strengths**: Handles paraphrasing, concept-level queries ("how do bones heal" → finds "fracture healing biology")
- **Backend**: Vector embeddings with FAISS index
- **When dominant**: Conceptual questions, clinical scenarios

### Signal 3: Hierarchical Tree Reasoning
- **Purpose**: Document structure-aware retrieval via Groq-generated document trees
- **Strengths**: Navigates textbook chapter → section → subsection hierarchy
- **Backend**: Tree nodes generated by Groq LLaMA, stored per document
- **When dominant**: Questions that map to specific textbook sections

### Fusion Strategy
All three signals are queried in parallel, results are fused using Reciprocal Rank Fusion (RRF), and the top-k chunks are injected into the Claude context window.

## Autonomous Librarian
The system automatically selects relevant documents from the uploaded library based on query intent. No manual document selection is needed. The librarian analyzes the query, identifies which uploaded PDFs are relevant, and routes retrieval accordingly.

## Confidence Scoring
Confidence is computed based on:
- **Retrieval relevance scores** from all three signals
- **Citation density** — how many chunks directly support the answer
- **Grounding classification** — whether evidence is direct, inferred, or absent

## Document Library
The platform supports PDF uploads of orthopaedic textbooks. Documents are:
1. Parsed into text chunks with page-level metadata
2. Indexed in BM25 (keyword) and FAISS (semantic) indices
3. Tree structures generated for hierarchical navigation
4. Available for cross-referencing in responses
